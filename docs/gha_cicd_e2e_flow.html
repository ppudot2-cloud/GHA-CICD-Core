⚡ Power Platform CI/CD — End-to-End Guide

This page explains the entire automated pipeline that takes a Power Platform solution from a developer's sandbox all the way to Production. Each section explains the why before the technical details — no prior DevOps knowledge required.

Two-Pipeline Architecture Zero Static Secrets (OIDC + AKV) 7 Environments ServiceNow CR Integration JFrog Artifactory
🗺️ Architecture Overview
The system uses two repositories and a two-pipeline design. GHA-Core is a shared toolbox containing all reusable build and deploy logic. GHA-Dynamics is the project-specific caller that provides configuration — which solutions to deploy, which URLs to use.
Two-Repo Design: All build/deploy scripts and reusable workflows live in GHA-Core (shared, org-wide). Only project config (solutions.json, deployment-settings, project-vars.yml) lives in GHA-Dynamics. This keeps the toolbox clean and shareable across multiple projects without code duplication.

  │                                                                              │

  │                          Open PR: feature/pipeline-N → main            │
  │                                                                              │
  │  ──── PR REVIEWED & MERGED TO MAIN ─────────────────────────────────────    │
  │                                                                              │
  │  PIPELINE 2  —  deploy-prod.yml  (GHA-Dynamics)                             │
  │  Auto-trigger: pipeline-context.json pushed to main at PR merge              │
  │                                                                              │
  │  Guard → Read Context → UAT re-validate → 🚦 Prod Approval Gate        │
  │                                                   ↓                         │
  │                                            Prod Deploy                      │
  └──────────────────────────────────────────────────────────────────────────────┘
Why two pipelines? Production needs a mandatory human gate. Pipeline 1 does all the work (build + test environments). Pipeline 2 fires only when someone reviews and merges the PR — providing an explicit human checkpoint before Production. The two are linked by pipeline-context.json, a handshake file that carries the Pipeline 1 build artifacts reference to Pipeline 2.
Zero Static Secrets: No Power Platform passwords or API keys are stored in GitHub Secrets. Instead, the pipeline logs into Azure using OIDC (a short-lived token generated per job), then retrieves credentials from Azure Key Vault at runtime. Credentials rotate independently and are never visible in logs.

🚀 Pipeline 1 — Build & Deploy
File: GHA-Dynamics/.github/workflows/build-and-deploy.yml
The main workhorse. Exports solutions from sandbox, packs ZIPs, runs Solution Checker, deploys to all pre-production environments, then opens a PR to queue Production.
Triggers:
• Manual (workflow_dispatch) — click "Run workflow" in GitHub Actions UI
• Automatic (push) — pushing commits to any feature/** branch fires this automatically, with skip_export set to true (source is already committed)
⚙️

▼

Rollback: Controlled by the ENABLE_ROLLBACK GitHub Environment variable (set per environment in Settings → Environments). When true, the pipeline takes a pre-import backup and auto-restores on failure. There is no enable_backup dispatch input — rollback behaviour is always driven by the environment variable.

🔍
Inside each Deploy job — Steps A through H
The deploy-all-solutions composite action orchestrates these steps in order for every environment
▼
  • A
    Reveille (Bootstrap)
    Checkout repos, Azure OIDC login, fetch PP credentials from Key Vault, merge global + project variables into GITHUB_ENV.
  • B
    PAC CLI Auth
    Authenticates PAC CLI against the target Dataverse environment using credentials loaded by Reveille.
  • C
    Pre-flight Checks (informational only)
    For each solution: verify artifact present → resolve deployment-settings tokens → base solutions check (warns if missing) → blocking check (warns if async ops in-progress) → version compare (logged for visibility). None of these checks ever block the pipeline — all results are warnings. skip_import is always false.
  • D
    ServiceNow: Open Change Request if enabled
    Creates a Change Request in ServiceNow. Pipeline waits until the CR is approved before proceeding to import. Only reached after all pre-flight checks pass.
  • E
    Deploy Loop — sequential per solution
    For each solution: find in target → backup if upgrading (when ENABLE_ROLLBACK=true is set on the GitHub Environment) → import solution (stage-and-upgrade or standard) → import config migration data Prod only → publish customizations → activate Cloud Flows + Classic Workflows → tag JFrog artifact Prod only → write per-solution summary card.
  • F
    ServiceNow: Close Change Request always, if enabled
    Closes the CR as successful or unsuccessful based on deploy outcome. Runs even if deploy failed.
  • G
    Post-deploy Verification
    Calls verify-deployment action — queries Dataverse to confirm each solution is installed at the expected version.
  • H
    Upload Backup Artifact + Structured Audit Log
    Uploads backup ZIPs as GitHub Artifacts (30-day retention). Writes a JSON audit log entry (environment, version, actor, timestamp, CR number, step outcomes) for compliance.
🏁 Pipeline 2 — UAT Re-validation + Production
File: GHA-Dynamics/.github/workflows/deploy-prod.yml
Fires automatically when Pipeline 1's PR is merged to main. Re-validates on UAT, then deploys to Production after a human approves.
Auto-triggered by PR merge: In normal flow, you never manually start Pipeline 2. Merging the PR created by Pipeline 1 is the trigger. The pipeline watches for pipeline-context.json being pushed to main — this happens at PR merge time.

⏪ GHA-Core: rollback.yml — Manual Rollback
File: GHA-Core/.github/workflows/rollback.yml — run from the GHA-Core repository Actions tab
Emergency restore of a Power Platform environment to a previous version using a backup artifact from a prior pipeline run. All rollback logic lives in GHA-Core. GHA-Dynamics has no rollback workflow.
Auto-rollback vs Manual rollback: When ENABLE_ROLLBACK=true is set on a GitHub Environment, deploy-all-solutions automatically backs up and restores on import failure — no manual intervention. This workflow is for post-deploy regressions where you need to roll back after a successful deploy.
⚠️ Safety gate: The confirm input must exactly match the environment name selected (e.g., type "Prod" when rolling back Prod). The validate job aborts immediately if they don't match — prevents accidental mis-fires.
⚙️
Inputs (workflow_dispatch)
▼
InputRequiredDescription
environmentYesTarget: Dev, Intg, UAT, FRS, Perf, Prod
environment_urlYesPower Platform environment URL (e.g. https://contoso.crm.dynamics.com)
source_repoYesGHA-Dynamics repo that produced the backup (format: org/repo)
source_run_idYesGitHub Actions run ID of the original deploy. Found in the URL: .../actions/runs/RUN_ID
backup_artifact_nameYesBackup artifact name, e.g. backup-UAT-v42. Listed in the Artifacts section of the original run.
azure_client_id / azure_tenant_id / azure_subscription_id / azure_key_vault_nameYesAzure OIDC identity inputs — operator pastes values from the target environment's App Registration
confirmYesType the environment name exactly (e.g. "Prod") — safety confirmation
mock_deployNoSimulate rollback without touching Dataverse
Backup artifact naming convention: backup-{Environment}-v{RunNumber} — e.g. backup-UAT-v42. Created during normal deploys when ENABLE_ROLLBACK=true on the GitHub Environment. Stored as GitHub Artifacts (30-day retention). Used by Invoke-Rollback.ps1.
Job flow: validate (confirm == environment check) → rollback (OIDC login → AKV fetch → PAC install → download artifact → Invoke-Rollback.ps1). Single rollback job — the environment URL is provided as an input, so there's no need for one job per environment.
🔍 GHA-Core: _reusable-lint.yml — Config Linter
File: GHA-Core/.github/workflows/_reusable-lint.yml
Called by build-and-deploy.yml as the lint-config job. Catches configuration mistakes before the export stage. Failures block pipeline progression. No Azure login needed — pure file-system validation.
Trigger: workflow_call only — called from build-and-deploy.yml lint-config job. Not directly dispatchable. Inputs: solutions_json_path (default solutions.json), project_vars_path (default .github/config/project-vars.yml).
📋
What gets validated (Python — no Azure login needed)
▼
  • 1
    solutions.json
    Valid JSON, required fields (name, folder, deployOrder, deploymentSettings), folder paths exist, deployOrder values are positive unique integers, dependsOn references resolve to declared solution names.
  • 2
    Deployment-settings JSONs
    All paths declared in solutions.json exist on disk, each file is valid JSON, has EnvironmentVariables and ConnectionReferences arrays. Paths not following the src/solutions/{Name}/deployment-settings-{env}.json convention emit a warning. Unresolved #{TOKEN}# placeholders are warned.
  • 3
    project-vars.yml
    Valid YAML. Does not contain protected keys (PP_CHECKER_GEO, PP_CHECKER_ERROR_LEVEL, DEFAULT_SOLUTION_TYPE, ENABLE_BACKUP, ENABLE_ROLLBACK). No credential-looking values (password, secret, token, key) in plaintext.
  • 4
    XML schema files
    Paths declared in solutions.json (dataSchemaFile) exist and are well-formed XML. Convention: src/solutions/{Name}/config-data-schema.xml.
✅ Errors vs Warnings: ::error:: annotations fail the job and block the pipeline. ::warning:: annotations are visible in the run summary but allow the pipeline to continue. Convention violations (wrong file paths) are warnings; missing files are errors.

Reusable action building blocks in GHA-Core. Each is a named sequence of steps callable from any job.

reveille — Bootstrap & Credential Fetch
GHA-Core/.github/actions/dynamics/reveille/action.yml
First step of every deploy/build job. Checks out repos, logs into Azure, fetches credentials from Key Vault, merges variables.
▼
Steps (in order)
  • 1
    Checkout caller repo
    Checks out GHA-Dynamics (full history for version stamping).
  • 2
    Checkout GHA-Core to .ci/
    Makes all GHA-Core scripts and actions available locally for the job.
  • 3
    OIDC Diagnostic skipped in mock
    Validates Azure identity inputs are non-empty and prints the OIDC subject claim. Essential for diagnosing AADSTS700016 (wrong Client ID) and AADSTS70021 (missing federated credential) errors.
  • 4
    Azure Login (OIDC) skipped in mock
    Logs into Azure using a temporary OIDC token — no stored password needed. Token is issued by GitHub and valid only for this job's duration.
  • 5
    Azure Key Vault secret fetch skipped in mock
    Fetches PP_APP_ID, PP_CLIENT_SECRET, PP_TENANT_ID from the project's Key Vault. Optionally fetches JFrog, MuleSoft, ServiceNow credentials. Each secret is masked with ::add-mask:: before being written to GITHUB_ENV.
  • 6
    Merge Variables
    Runs Merge-Variables.ps1: reads global-vars.yml (GHA-Core) + project-vars.yml (GHA-Dynamics), merges (project overrides non-protected globals), writes all to GITHUB_ENV for use by all subsequent steps. Exits 1 if project tries to override a protected key.

📊 Variable Precedence — Which Value Wins?
Variables can be set at multiple levels. The highest-priority level always wins. Think of it as a layered override system — each layer adds or overrides the one below it.
The Golden Rule: Values flow upward from GHA-Core (base defaults) through to Azure Key Vault (highest priority). A variable set at a higher layer always overrides the same variable at a lower layer. Exception: protected keys are locked at layer 1 and cannot be overridden by anyone.
5 — WINS

PP credentials, JFrog key, ServiceNow creds. Fetched at runtime via OIDC. Masked immediately. Stored nowhere in GitHub.

🔒 Protected Keys (locked at layer 1 — cannot be overridden by any project):
PP_CHECKER_GEO · PP_CHECKER_ERROR_LEVEL · DEFAULT_SOLUTION_TYPE · ENABLE_BACKUP · ENABLE_ROLLBACK

If project-vars.yml contains any of these, Merge-Variables.ps1 exits with error code 1 and the pipeline fails immediately. The _reusable-lint.yml workflow also catches this before the pipeline starts.
Azure Identity Keys are intentionally excluded from the merge: AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID, AZURE_KEY_VAULT_NAME are passed directly as workflow inputs from vars.* — they bypass the variable merge entirely. This prevents OIDC hijacking: a compromised project-vars.yml cannot redirect authentication to a different Azure tenant.

Precedence in practice — example

  Variable: SOLUTION_IMPORT_MAX_WAIT_MINUTES

  Layer 1 global-vars.yml    : 60      (base default)
  Layer 2 project-vars.yml   : 90      (project override — larger solutions need more time)
  Layer 3 GitHub repo vars   : (not set)
  Layer 4 GitHub env vars    : (not set)
  Layer 5 AKV               : (not a secret — not applicable)

  ► Effective value: 90  (project-vars.yml wins over global default)

  ─────────────────────────────────────────────────────

  Variable: PP_CHECKER_GEO

  Layer 1 global-vars.yml    : "UnitedStates"  (PROTECTED)
  Layer 2 project-vars.yml   : "Europe"        ← ERROR: Merge-Variables.ps1 exits 1
  Layer 3–5                  : (irrelevant — merge fails immediately)

  ► Pipeline FAILS with: "Cannot override protected key PP_CHECKER_GEO"

✅ Why Key Vault instead of GitHub Secrets for PP credentials? GitHub Secrets are scoped to one repo and require manual rotation with GitHub UI access. Azure Key Vault secrets: rotate independently of GitHub (no PR, no deploy needed), have their own audit log in Azure Monitor, can be scoped per environment (non-prod KV vs prod KV), and are fetched via OIDC — meaning zero stored passwords are needed to access the vault itself.
⚠️ OIDC Federated Credential Setup: For each GitHub Environment that runs deploys (Dev, Intg, UAT, FRS, Perf, Prod), the App Registration must have a federated credential with:
• Issuer: https://token.actions.githubusercontent.com
• Subject: repo:ppudot2-cloud/GHA-Dynamics:environment:{EnvName}
Without this, Azure login fails with AADSTS70021 ("No matching federated identity record"). See OIDC_SETUP.md for the full setup guide.
